### Таблицы (17 таблиц)

#### Затраты Иридиум (4 таблицы):
1. **SPNET_TRAFFIC** - данные трафика SPNet
2. **STECCOM_EXPENSES** - расходы STECCOM
3. **TARIFF_PLANS** - тарифы Иридиум с ценами превышения
4. **IRIDIUM_SERVICES_INFO** - информация о сервисах (импорт из VIEW)

#### Биллинг - доходы (7 таблиц):
5. **SERVICES** - основная таблица услуг биллинга
6. **BM_TARIFF** - справочник тарифов для клиентов
7. **BM_SERVICE_TARIFF** - связь услуг с тарифами по периодам
8. **BM_TARIFFEL** - тарифные элементы с ценами (MONEY)
9. **BM_TYPE** - справочник типов услуг
10. **BM_INVOICE** - счета-фактуры (накладные) для выставления счетов клиентам. Формируются на основе ANALYTICS. 1 ЧИСЛА данные переносятся из ANALYTICS в счета-фактуры, что ЗАКРЫВАЕТ ФИНАНСОВЫЙ ПЕРИОД. После формирования счета-фактуры уходят в систему 1С (связь через OUTER_IDS по CUSTOMER_ID, где TBL = 'CUSTOMERS')
11. **BM_INVOICE_ITEM** - детали счетов-фактур с суммами начислений (MONEY). Формируется на основе ANALYTICS. 1 ЧИСЛА данные переносятся из ANALYTICS в BM_INVOICE_ITEM, что закрывает финансовый период. После формирования счета-фактуры уходят в систему 1С
12. **ANALYTICS** - таблица аналитики, ОСНОВА для формирования счетов-фактур. Формируется на основе BM_SERVICE_MONEY. ОБНОВЛЯЕТСЯ КАЖДЫЙ ДЕНЬ НЕСКОЛЬКО РАЗ на основе сессий. Содержит ВСЕ начисления, включая тесты и эксперименты. НЕ ВСЕ записи попадают в счета-фактуры (INVOICE_ITEM_ID IS NULL для тестов). 1 ЧИСЛА данные по договорам переносятся в счета-фактуры, что закрывает финансовый период
13. **BM_PERIOD** - справочник финансовых периодов (ВАЖНО: таблица называется BM_PERIOD, не BM_PERIODS!). Для Иридиум: периоды с 2 числа по 1 число следующего месяца. Колонки: START_DATE, STOP_DATE (не DATE_BEG/DATE_END!), MONTH (не PERIOD_NAME!)
18. **CUSTOMERS** - таблица клиентов биллинга. ВАЖНО: НЕТ колонок CODE_1C, ORGANIZATION_NAME, PERSON_NAME! CODE_1C получается через OUTER_IDS.EXT_ID (где TBL='CUSTOMERS'). Название организации/ФИО получается через BM_CUSTOMER_CONTACT
19. **ACCOUNTS** - таблица договоров клиентов. ВАЖНО: НЕТ колонки ACCOUNT_NAME! Название договора хранится в DESCRIPTION
14. **BM_RESOURCE_TYPE** - справочник типов ресурсов (расшифровки услуг) для бухгалтерии. MNEMONIC имеет хороший мнемонический смысл
15. **BM_SERVICE_MONEY** - сессии услуг с объемами трафика и начислениями. ОСНОВА для формирования ANALYTICS. Зависит от BM_CDR_ACCT для SBD услуг
16. **BM_CDR_ACCT** - таблица трафика (CDR). Источник данных о трафике для BM_SERVICE_MONEY. Для SBD услуг используется BM_CDR_ACCT, RADACCT НЕ используется
17. **BM_ZONE** - справочник тарифных зон. Важна для понимания тарифных зон с фиксированной ценой зоны в тарифном плане. Используется в BM_TARIFF (PH_ZONE_GROUP_ID, TR_ZONE_GROUP_ID) и BM_TARIFFEL (ZONE_ID). ВАЖНО: НЕТ колонки ZONE_NAME! Используйте DESCRIPTION как название зоны
18. **BM_PERIOD** - справочник финансовых периодов (ВАЖНО: таблица называется BM_PERIOD, не BM_PERIODS!). Для Иридиум: периоды с 2 числа по 1 число следующего месяца. Колонки: START_DATE, STOP_DATE (не DATE_BEG/DATE_END!), MONTH (не PERIOD_NAME!)
19. **CUSTOMERS** - таблица клиентов биллинга. ВАЖНО: НЕТ колонок CODE_1C, ORGANIZATION_NAME, PERSON_NAME! CODE_1C получается через OUTER_IDS.EXT_ID (где TBL='CUSTOMERS'). Название организации/ФИО получается через BM_CUSTOMER_CONTACT
20. **ACCOUNTS** - таблица договоров клиентов. ВАЖНО: НЕТ колонки ACCOUNT_NAME! Название договора хранится в DESCRIPTION

#### Себестоимость (концептуальная):
21. **COST_OF_GOODS_SOLD** - концептуальная таблица для расчета себестоимости. Не является физической таблицей. Представляет логику расчета себестоимости путем сопоставления доходов (ANALYTICS, BM_INVOICE_ITEM) с затратами Иридиум (STECCOM_EXPENSES, SPNET_TRAFFIC) по SUB-XXXXX. ВАЖНО: Финансистов волнуют ВСЕ SUB-, даже тестовые. Иридиум не всегда выставляет счета по тестовым услугам. SQL запрос: oracle/queries/cost_of_goods_sold_analysis.sql

## 📝 Формат файлов

### Таблицы/VIEW (JSON)
Каждый файл содержит:
- `table_name` / `view_name` - название
- `schema` - схема БД
- `description` - описание назначения
- `database` - тип БД (Oracle/PostgreSQL)
- `ddl` - DDL statement (для таблиц)
- `key_columns` - ключевые колонки с описаниями
- `business_rules` - бизнес-правила
- `relationships` - связи с другими таблицами
- `usage_notes` - примечания по использованию

### Q/A примеры (JSON)
Массив объектов с полями:
- `question` - вопрос на естественном языке
- `sql` - SQL запрос
- `context` - контекст запроса
- `business_entity` - бизнес-сущность
- `complexity` - сложность (1-4)
- `category` - категория

## 🔄 Процесс наполнения

База знаний создана как основа и будет постепенно наполняться:

1. **Уточнение описаний** - добавление деталей по мере изучения бизнес-логики
2. **Дополнение Q/A** - добавление новых примеров запросов
3. **Корректировка связей** - уточнение relationships между таблицами
4. **Добавление бизнес-правил** - документирование логики расчетов
5. **Создание новых VIEW** - добавление описаний для новых представлений

## 📋 Следующие шаги

При создании нового отчета по себестоимости:
1. Изучить структуру данных для расчета доходов
2. Определить связь между нашими тарифами и тарифами Иридиум
3. Создать VIEW для расчета себестоимости
4. Добавить описание нового VIEW в базу знаний
5. Добавить Q/A примеры для нового отчета
