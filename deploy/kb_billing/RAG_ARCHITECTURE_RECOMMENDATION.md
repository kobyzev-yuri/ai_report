# Рекомендации по RAG архитектуре для KB_billing

## Анализ структуры данных

### Текущая структура KB_billing

**Типы данных:**
1. **DDL описания** - структурированные SQL схемы (17 таблиц + 4 представления)
2. **Документация** - текстовые описания таблиц, колонок, бизнес-правил (JSON файлы)
3. **Q/A примеры** - 24 пары вопрос (русский) → SQL запрос
4. **Метаданные** - связи между таблицами, домены, категории

**Характер связей между таблицами:**
- Иерархическая цепочка: `BM_CDR_ACCT → BM_SERVICE_MONEY → ANALYTICS → BM_INVOICE_ITEM → BM_INVOICE`
- Связи по ключам: `CONTRACT_ID` (SUB-XXXXX), `IMEI`, `SERVICE_ID`
- Простые JOIN операции (LEFT JOIN, JOIN)
- Связи документированы в JSON файлах как `relationships`

**Примеры связей:**
```json
{
  "table": "SPNET_TRAFFIC",
  "type": "LEFT JOIN",
  "on": "SERVICES.LOGIN = SPNET_TRAFFIC.CONTRACT_ID AND SERVICES.VSAT = SPNET_TRAFFIC.IMEI"
}
```

## Рекомендация: **Плоская база с similarity search ДОСТАТОЧНА**

### ✅ Почему плоская база подходит:

#### 1. **Простота связей**
- Связи между таблицами простые (JOIN по ключам)
- Нет сложных многоуровневых графовых зависимостей
- Связи уже документированы в JSON файлах (`relationships`)
- LLM может понять связи из документации без графовой навигации

#### 2. **Характер задач**
- **Основная задача**: семантический поиск похожих примеров SQL
- **Вторичная задача**: поиск DDL и документации по таблицам
- **Не требуется**: навигация по сложным графам зависимостей

#### 3. **Объем данных**
- 17 таблиц + 4 представления (умеренный объем)
- 24 Q/A примера (можно расширить)
- Связи документированы явно в JSON

#### 4. **Эффективность**
- Плоская база проще в реализации и поддержке
- Быстрее поиск (один запрос вместо обхода графа)
- Меньше сложности в интеграции

## Рекомендуемая архитектура RAG

### Вариант 1: **Гибридный поиск (Рекомендуется)**

```
┌─────────────────────────────────────────┐
│         Финансист (вопрос)              │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│      AI Agent (LLM + RAG)               │
│  ┌──────────────────────────────────┐   │
│  │  1. Понимание запроса            │   │
│  │  2. Гибридный поиск:             │   │
│  │     a) Семантический (векторный) │   │
│  │     b) Фильтры по метаданным     │   │
│  │  3. Извлечение связей из JSON   │   │
│  │  4. Генерация SQL                │   │
│  └──────────────────────────────────┘   │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│   Векторная БД (Qdrant)                 │
│  ┌──────────────────────────────────┐   │
│  │  Collection: kb_billing          │   │
│  │  ├── DDL (структура БД)          │   │
│  │  ├── Documentation (описания)    │   │
│  │  ├── Q/A Examples (примеры SQL)   │   │
│  │  └── Relationships (связи)      │   │
│  │                                   │   │
│  │  Payload метаданные:              │   │
│  │  - type: "ddl"|"doc"|"qa"|"rel"  │   │
│  │  - table_name: "SPNET_TRAFFIC"   │   │
│  │  - category: "Превышение трафика"│   │
│  │  - complexity: 1-4               │   │
│  │  - relationships: [...]          │   │
│  └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
```

**Преимущества:**
- ✅ Семантический поиск для похожих примеров
- ✅ Фильтрация по метаданным (таблица, категория, сложность)
- ✅ Связи хранятся в payload и извлекаются при необходимости
- ✅ Простая реализация и поддержка

### Структура данных в векторной БД

#### Тип 1: DDL (структура таблиц)
```json
{
  "id": "ddl_spnet_traffic",
  "vector": [...],
  "payload": {
    "type": "ddl",
    "table_name": "SPNET_TRAFFIC",
    "content": "CREATE TABLE SPNET_TRAFFIC (...)",
    "schema": "billing",
    "key_columns": {...},
    "relationships": [
      {
        "table": "SERVICES",
        "type": "LEFT JOIN",
        "on": "SERVICES.LOGIN = SPNET_TRAFFIC.CONTRACT_ID"
      }
    ]
  }
}
```

#### Тип 2: Документация
```json
{
  "id": "doc_spnet_traffic",
  "vector": [...],
  "payload": {
    "type": "documentation",
    "table_name": "SPNET_TRAFFIC",
    "content": "Таблица трафика SPNet - данные об использовании Iridium M2M сервисов...",
    "business_rules": [...],
    "usage_notes": [...]
  }
}
```

#### Тип 3: Q/A примеры
```json
{
  "id": "qa_example_001",
  "vector": [...],
  "payload": {
    "type": "qa_example",
    "question": "Покажи превышение трафика за октябрь 2025",
    "sql": "SELECT ... FROM v_consolidated_report_with_billing ...",
    "category": "Превышение трафика",
    "complexity": 2,
    "business_entity": "overage",
    "table_names": ["V_CONSOLIDATED_REPORT_WITH_BILLING"]
  }
}
```

#### Тип 4: Relationships (связи)
```json
{
  "id": "rel_spnet_services",
  "vector": [...],
  "payload": {
    "type": "relationship",
    "source_table": "SPNET_TRAFFIC",
    "target_table": "SERVICES",
    "join_type": "LEFT JOIN",
    "join_condition": "SERVICES.LOGIN = SPNET_TRAFFIC.CONTRACT_ID AND SERVICES.VSAT = SPNET_TRAFFIC.IMEI",
    "description": "Связь трафика SPNet с услугами биллинга"
  }
}
```

## Стратегия поиска

### Шаг 1: Семантический поиск
```python
# Поиск похожих примеров по вопросу финансиста
results = vector_db.search(
    query_vector=embed_question("Покажи превышение трафика за октябрь"),
    query_filter={
        "must": [
            {"key": "type", "match": {"value": "qa_example"}},
            {"key": "category", "match": {"value": "Превышение трафика"}}
        ]
    },
    limit=5
)
```

### Шаг 2: Извлечение связей
```python
# Извлечение связей для найденных таблиц
table_names = extract_table_names(results)
relationships = vector_db.search(
    query_filter={
        "must": [
            {"key": "type", "match": {"value": "relationship"}},
            {"key": "source_table", "match": {"any": table_names}}
        ]
    }
)
```

### Шаг 3: Формирование контекста
```python
context = {
    "qa_examples": results,
    "relationships": relationships,
    "ddl": get_ddl_for_tables(table_names),
    "documentation": get_docs_for_tables(table_names)
}
```

### Шаг 4: Генерация SQL
```python
sql = llm.generate_sql(
    question=question,
    context=context
)
```

## Когда Graph KB может быть полезен?

Graph KB имеет смысл, если:

### ❌ НЕ нужен Graph KB, если:
- Связи простые (JOIN по ключам) ✅ **Ваш случай**
- Связи документированы явно ✅ **Ваш случай**
- Основная задача - поиск примеров ✅ **Ваш случай**
- Умеренный объем данных ✅ **Ваш случай**

### ✅ Graph KB полезен, если:
- Сложные многоуровневые зависимости (например, цепочки из 5+ таблиц)
- Неявные связи, которые нужно выводить
- Требуется навигация по графу (например, "найди все таблицы, связанные с ANALYTICS через 2 уровня")
- Большой объем данных с множеством связей
- Требуется анализ влияния изменений (impact analysis)

## Рекомендации по реализации

### 1. Векторная БД

**Используется Qdrant** (коллекция `kb_billing`, настройки в config.env: QDRANT_HOST, QDRANT_PORT, QDRANT_COLLECTION). См. [docs/kb-billing-vs-presales.md](../../docs/kb-billing-vs-presales.md).

### 2. Модель эмбеддингов

**Рекомендация:** `intfloat/multilingual-e5-large` (1024d)
- Отличная поддержка русского языка
- Хорошее качество для SQL и документации

**Альтернатива:** `cointegrated/rubert-tiny2` (312d) - легковесный вариант

### 3. Структурирование данных

**Чанкинг:**
- DDL: по таблице (1 чанк = 1 таблица)
- Документация: по таблице (1 чанк = 1 таблица)
- Q/A примеры: по примеру (1 чанк = 1 пример)
- Relationships: по связи (1 чанк = 1 связь)

**Метаданные:**
- Обязательные: `type`, `table_name`, `category`
- Опциональные: `complexity`, `business_entity`, `relationships`

### 4. Гибридный поиск

**Комбинация:**
1. **Векторный поиск** - семантическое сходство
2. **Фильтры по метаданным** - точное соответствие
3. **Извлечение связей** - из payload relationships

**Пример:**
```python
# Гибридный поиск
results = vector_db.search(
    query_vector=embedding,
    query_filter={
        "must": [
            {"key": "type", "match": {"value": "qa_example"}},
            {"key": "category", "match": {"value": "Превышение трафика"}}
        ],
        "should": [
            {"key": "table_names", "match": {"any": ["V_CONSOLIDATED_REPORT_WITH_BILLING"]}}
        ]
    },
    limit=5
)
```
