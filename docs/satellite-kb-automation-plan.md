# План: автоматизация построения спутникового домена базы знаний

Цель: заложить знание о **типичных документах** спутникового домена и использовать его для автоматизации наполнения KB (Confluence, схемы, инструкции, таблицы) и улучшения работы спутникового ассистента и библиотекаря.

---

## 1. Типичные типы документов в спутниковом домене

| Тип документа | Описание | Форматы / источники | Парсер | Примеры |
|---------------|----------|----------------------|--------|---------|
| **network_scheme** | Схема сети (топология, узлы, связи) | PNG, JPG, PDF со схемами | image + Vision, pdf | Схемы BGP, VPWS, доступа к NMS |
| **device_list** | Список устройств, конфигурации | XLS, XLSX | xls, xlsx | net devices list, списки терминалов |
| **instruction** | Инструкция, руководство | DOCX, PDF | docx, pdf | Инструкции ЦУС, мобильные МЧС |
| **diagram_drawio** | Диаграмма draw.io (редактируемая) | .drawio, .xml (drawio) | drawio + Gemini | Схемы подключения VSAT |
| **spreadsheet** | Таблицы, отчёты | XLS, XLSX | xls, xlsx | Конфиги, расчёты |
| **page_content** | Текст страницы Confluence | body.storage | storage parser | Описания, пояснения на странице |

Служебные форматы Confluence (не индексировать): **.tmp**, **.render**, **.tfss**.

---

## 2. Конфигурация «тип документа → парсер и метаданные»

Предлагается ввести конфиг (например `kb_billing/rag/satellite_doc_types.yaml` или секция в существующем конфиге):

- **По расширению / MIME**: какой парсер применять (уже есть в `attachment_parsers`).
- **По паттерну имени или страницы** (опционально): например, `*net*devices*`, `*BGP*` → пометить как `document_type: device_list` или `network_scheme` для улучшения поиска и подсказок ассистента.
- **Метаданные в чанках**: при загрузке в Qdrant сохранять в payload поле `document_type` (network_scheme, device_list, instruction, diagram_drawio, spreadsheet, page_section), чтобы ассистент и фильтры могли опираться на тип.

Шаги реализации:

1. Добавить в `confluence_kb_generator` (или в общий слой загрузки) опциональное определение `document_type` по имени файла/страницы (правила из конфига или хардкод типов по расширению).
2. При формировании документа KB записывать `document_type` в секцию/source.
3. В `kb_loader.load_confluence_docs()` при создании точки Qdrant пробрасывать `document_type` в payload.

## 3. Автоматизация синхронизации пространства

- **Анализ перед синхронизацией**: скрипт `analyze_confluence_space.py` уже даёт сводку по типам вложений и парсерам. Использовать его вывод (или вызов из UI) перед массовой синхронизацией, чтобы видеть, какие парсеры понадобятся и что будет пропущено (skip/unsupported).
- **Пакетная синхронизация**: в UI библиотекаря — возможность задать список пространств (например `~a.zhegalov`, `SPC`) и по кнопке запустить синхронизацию по очереди с лимитом страниц на пространство. Результат — несколько JSON в `confluence_docs/` (по одному на пространство или один объединённый — по решению).
- **Профиль пространства** (опционально): при выборе пространства указывать профиль — «инженерный» (схемы, device list, draw.io), «инструкции» (DOCX, PDF), «смешанный». Профиль влияет на подсказки и по умолчанию на то, какие типы считать приоритетными при лимитах.

## 4. Роль спутникового ассистента и библиотекаря

- **Знание о типичных документах**: в системный промпт ассистента добавить краткое описание типов (network_scheme, device_list, instruction, diagram_drawio, spreadsheet) и формулировку вроде: «В KB могут быть схемы сети (картинки/PDF), списки устройств (XLS/XLSX), инструкции (DOCX/PDF), draw.io-диаграммы. Используй тип документа из контекста, если он указан.»
- **Подсказки при поиске**: если пользователь спрашивает про «список устройств» или «схему BGP», ассистент может упоминать, что такие данные часто лежат в вложениях типа device_list или network_scheme и при необходимости подсказать, как сузить поиск (по названию страницы, по типу вложения).
- **Рекомендации по наполнению KB**: библиотекарь при выборе пространства/страниц может показывать краткую сводку из анализа (как сейчас в `analyze_confluence_space`) и рекомендовать: «Здесь 15 изображений (будут через Vision), 1 XLS — убедитесь, что xlrd установлен; 9 без расширения — проверьте, не draw.io ли по mediaType.»

## 5. Очистка и удаление неудачных попыток из KB (только спутниковый домен)

- **outdated.txt**: уже есть — список `page_id`, которые не загружаются в Qdrant при `reload_confluence_only()`. Для удаления «мусорных» попыток: добавить нужные `page_id` в `confluence_docs/outdated.txt` и перезапустить «Перезагрузить в Qdrant только Confluence». Биллинг не трогается.
- **Удаление документов из JSON**: при необходимости удалить целый документ (страницу) из KB — удалить его из соответствующего JSON в `confluence_docs/` (или заменить на пустой content) и снова выполнить перезагрузку только Confluence. Либо скрипт/кнопка «Удалить из KB по page_id»: убрать документ из JSON и добавить page_id в outdated.txt.
- **Не перезагружать биллинг**: все операции только над `type=confluence_section` и источниками из `confluence_docs/*.json`; коллекция Qdrant для биллинга (Q/A, таблицы, представления) не изменяется.

---

## 6. Порядок внедрения (краткий roadmap)

1. **XLS/XLSX** — парсеры добавлены; зависимость `xlrd` в requirements; анализ пространства уже помечает .xls/.xlsx как обрабатываемые.
2. **Метаданные document_type** — опционально в конфиге; вывод типа по расширению (image→network_scheme/diagram, xls/xlsx→device_list/spreadsheet, docx/pdf→instruction); запись в payload чанка в `kb_loader`.
3. **Конфиг типов документов** — YAML/JSON с правилами (расширение, паттерн имени → document_type); загрузка в генераторе и в loader.
4. **UI**: пакетная синхронизация по списку пространств; опционально «профиль пространства»; показ сводки анализа перед синхронизацией (или ссылка на запуск `analyze_confluence_space`).
5. **Ассистент**: обновить системный промпт описанием типов документов; при ответе использовать document_type из контекста, если есть.
6. **Очистка KB**: кнопка или скрипт «Исключить из KB по page_id» (добавить в outdated + при желании убрать из JSON).

После этого спутниковый домен KB будет опираться на явное знание о типичных документах и парсерах, а наполнение и очистка — более предсказуемыми и управляемыми.
