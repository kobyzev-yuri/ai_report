# Что попадает в KB при интеграции с Confluence

## Кратко

В базу знаний (Qdrant) попадает **весь контент страницы**:

1. **Тело страницы** — HTML из редактора Confluence (заголовки, параграфы, списки, таблицы).
2. **Вложения** — для каждого вложения страницы:
   - **PDF** — извлечение текста (PyMuPDF);
   - **DOCX/DOC** — извлечение текста и таблиц (python-docx);
   - **Схемы сети Draw.io** (`.drawio`, `.drawio.xml`) — парсинг XML, извлечение подписей из `mxCell` (узлы, блоки); поддерживается сжатый формат.
   - **Изображения** (PNG, JPG и т.д.) — OCR (pytesseract + Tesseract), при отсутствии OCR — пометка «изображение/схема».

Текст из вложений добавляется в документ как секции «Вложение: имя_файла» и индексируется вместе со страницей. Таким образом, содержимое PDF (в т.ч. инструкции про iDirect X5 и т.п.) участвует в поиске по KB.

В каждом документе в JSON сохраняется поле **attachments_processed**: список по каждому вложению страницы — `filename`, `status` (`indexed` / `skipped` / `error`), при успехе — `text_length`, при ошибке — `reason`. По нему видно, какие файлы реально проиндексированы, а какие пропущены или не скачались.

---

## Как устроена выгрузка

1. **Страницы** — Confluence REST API, `body.storage` → парсинг HTML (h1–h6, p, ul, ol, table) в структуру секций.
2. **Вложения** — для каждой страницы запрос `GET /rest/api/content/{pageId}/child/attachment`, скачивание по `_links.download`.
3. **Парсеры** (`kb_billing/rag/attachment_parsers.py`):
   - PDF → PyMuPDF (`fitz`), текст по страницам;
   - DOCX → python-docx, параграфы и таблицы;
   - **Draw.io** → парсинг XML (в т.ч. сжатый mxfile/diagram), сбор всех атрибутов `value` и текста из `mxCell` — подписи узлов и блоков схемы сети;
   - Изображения → PIL + pytesseract (OCR, рус+англ); при ошибке/отсутствии OCR — короткая пометка в тексте.
4. В генераторе (`confluence_kb_generator.py`) к секциям тела страницы добавляются секции «Вложение: filename» с извлечённым текстом. Итоговый `content[]` сохраняется в JSON. При загрузке в Qdrant (`kb_loader.py`) каждая секция (в т.ч. каждая «Вложение: …») становится отдельной точкой (тип `confluence_section`), чтобы поиск возвращал наиболее релевантные фрагменты, а не целую страницу.

## Настройки и ограничения

- **Включение/выключение вложений:** в `ConfluenceKBGenerator(index_attachments=True)` — по умолчанию включено.
- **Лимит размера вложения:** `max_attachment_size` (по умолчанию 50 MB) — файлы больше не скачиваются/не парсятcя, в документ пишется пометка.
- **Лимит длины текста из вложения:** `max_attachment_text_length` (по умолчанию 500 000 символов) — обрезка с пометкой «текст обрезан по лимиту».

## Зависимости

- **PDF:** `pymupdf`
- **DOCX:** `python-docx`
- **Изображения (OCR):** `Pillow`, `pytesseract`; в системе должен быть установлен **Tesseract OCR** (для русского: `tesseract-ocr-rus` или пакет с языком `rus`).

Если Tesseract не установлен или OCR недоступен, для изображений в документ попадёт пометка «изображение/схема — OCR не выполнен», без падения конвейера.

## Схемы сети (Draw.io)

Под «картинками» в контексте документации понимаются в т.ч. **схемы сети в формате Draw.io** (`.drawio`, `.drawio.xml`). Они обрабатываются отдельным парсером: из XML извлекаются все текстовые подписи элементов `mxCell` (названия узлов, блоков, соединений), так что по ним можно искать в KB без OCR. Поддерживается и сжатый вариант файла (base64 + deflate в `<diagram>`).

## Аннотация изображений (будущее)

Обычные растровые изображения (PNG, JPG) обрабатываются через OCR. Для качественного описания схем в виде картинок (не .drawio) можно позже подключить vision API и подмешивать описание в секцию «Вложение: …». Точка расширения — `extract_text_from_image` в `attachment_parsers.py`.

## Спутниковый ассистент (агент на Gemini)

Формированием и курированием базы знаний занимается **Спутниковый ассистент** — агент на базе Gemini с персоной квалифицированного инженера по подключению сетей заказчика к спутниковому сегменту. Он подсказывает, что индексировать, отвечает на вопросы по документации (с подмешиванием релевантных чанков из KB) и даёт рекомендации по структуре KB. Модель задаётся через `GEMINI_MODEL` в config.env (например `gemini-2.0-flash`, `gemini-2.5-pro`), ключ — `GEMINI_API_KEY` или `GOOGLE_API_KEY`. Код: `kb_billing/rag/satellite_librarian_agent.py`, интерфейс — блок «Спутниковый ассистент (Gemini)» в закладке «Спутниковый ассистент».

## Где это в коде

- **Клиент Confluence:** `kb_billing/rag/confluence_client.py` — `get_page_attachments()`, `download_attachment()`.
- **Парсеры вложений:** `kb_billing/rag/attachment_parsers.py` — `parse_attachment()`, `extract_text_from_pdf()`, `extract_text_from_docx()`, `extract_text_from_image()`.
- **Генератор KB:** `kb_billing/rag/confluence_kb_generator.py` — `_process_attachments()`, в `page_to_kb_doc()` к секциям добавляются вложения.
- **Загрузка в Qdrant:** `kb_billing/rag/kb_loader.py` — чанки по секциям (тип `confluence_section`).
- **Агент библиотекаря:** `kb_billing/rag/satellite_librarian_agent.py` — Gemini-агент с RAG по Confluence KB.
