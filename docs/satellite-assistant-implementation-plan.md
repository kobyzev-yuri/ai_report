# План имплементации: Спутниковый ассистент (KB как у эксперта + медиа + генерация схем)

**Отслеживание выполнения:** [satellite-assistant-implementation.md](satellite-assistant-implementation.md) — документ реализации с чеклистом этапов и отметками выполненных задач.

Референс: опыт проекта **~brats** (нейропродажник коттеджного посёлка на базе Gemini + мультимодальная KB). Цель — перенести лучшие практики на **Спутникового ассистента** (инженер присэйла): заполнение KB смысловыми блоками с аннотациями картинок/видео/презентаций, выдача ссылок на медиа при консультации, образцы draw.io в KB и в перспективе **генерация схем сети** по требованиям клиента.

---

## 1. Цели и референс (brats)

### Что хорошо в brats

| Аспект | Описание |
|--------|----------|
| **Заполнение KB** | Контент сайта и документов разбивается на **смысловые блоки** (semantic blocks); каждый блок — отдельная единица в KB с заголовком, текстом, привязкой к разделу. |
| **Аннотации медиа** | При добавлении в KB генерируются **аннотации к картинкам, видео, презентациям** (через vision/LLM): описание, alt, подпись для клиента. |
| **Консультация** | Агент может **давать ссылки на картинки** (на сайте или в специальной папке) и документы; в ответе возвращается структура `media: { images: [...], documents: [...] }`, чат отображает превью и кнопки «Скачать». |
| **Структура метаданных** | В чанках KB: `metadata.images[]` (url, description, alt), `metadata.documents[]` (url, title, description); поиск возвращает релевантные чанки, из них собираются медиа для ответа. |

### Что нужно Спутниковому ассистенту

- Те же принципы: **смысловые блоки** при индексации Confluence (не только «секция по заголовку», но и выделение логических блоков: описание решения, конфигурация, чек-лист, схема).
- **Аннотации** к изображениям и к **схемам draw.io** (сейчас draw.io только парсится в текст подписей mxCell; нужно описание «что за схема» для поиска и ответа).
- При ответе инженеру — **ссылки на релевантные схемы/документы** (Confluence, или статическая папка для «образцов для ознакомления»).
- **Образцы draw.io** в KB: типовые схемы сети (iDirect, Kingsat, топологии) как отдельные сущности с метаданными и возможностью «показать ссылку» или «открыть в draw.io».
- **Идеал:** генерация **новых схем сети** (draw.io-подобный вывод) по текстовому описанию требований клиента — работа присэйла.

## 2. Текущее состояние (ai_report)

- **Confluence → KB:** страницы парсятся по заголовкам (h1–h6) в секции; вложения обрабатываются: PDF, DOCX, **draw.io** (XML, подписи mxCell), изображения (OCR). Текст из вложений добавляется как секции «Вложение: filename». См. `docs/confluence-kb-what-is-indexed.md`.
- **Qdrant:** одна коллекция `kb_billing`; точки типа `confluence_section` с полями `content`, `metadata` (title, source_url, page_id и т.д.). **Нет** отдельного хранения `metadata.images` / `metadata.documents` в структуре, как в brats.
- **Спутниковый ассистент (Gemini):** чат с RAG: поиск по запросу → подмешивание чанков в контекст → ответ. Ссылки на Confluence по `source_url` в чанках уже показываются в expander «Какие чанки использованы». Отдельного блока «релевантные картинки/документы» с URL нет.
- **Draw.io:** парсинг есть (`attachment_parsers.extract_text_from_drawio`); в KB попадает текст подписей. Нет: аннотация «что за схема», хранение URL/файла схемы в метаданных для выдачи ссылки.

## 3. Фаза A: Заполнение KB — смысловые блоки и аннотации медиа

### 3.1. Смысловые блоки при индексации Confluence

**Цель:** не только «секция по заголовку», но и разбиение длинных текстов на логические блоки (как в brats: описание, конфигурация, чек-лист, предупреждения).

**Шаги:**

1. **Определить типы блоков** для спутниковой документации (по аналогии с brats):
   - `concept` — описание технологии/решения;
   - `howto` — пошаговая инструкция;
   - `config` — конфигурация, параметры;
   - `checklist` — чек-лист для инженера;
   - `warning` — ограничения, типовые ошибки;
   - `diagram` — схема (draw.io или изображение).
2. **В генераторе** (`confluence_kb_generator.py`): после `_parse_storage_to_content()` добавить (опционально) шаг **разбиения крупных секций** на смысловые блоки:
   - либо эвристики (длина, наличие списков, ключевые слова);
   - либо вызов LLM (Gemini) для разметки блоков по типам — как в brats `SemanticBlocksExtractor` / библиотекарь.
3. **Метаданные чанка:** сохранять `metadata.semantic_block_type`, `metadata.section` (уже есть аналог по заголовкам). При загрузке в Qdrant (`kb_loader.py`) передавать эти поля в `payload`.

**Отладка:**

- Юнит-тест: на одной странице Confluence с известной структурой проверить, что получаются блоки с нужными типами.
- В Streamlit после «Синхронизация» просматривать сохранённый JSON документа: наличие полей блоков и типов.

### 3.2. Аннотации к изображениям и draw.io

**Цель:** у каждого изображения и у каждой схемы draw.io в KB — краткое описание (аннотация) для поиска и для ответа клиенту.

**Шаги:**

1. **Изображения (PNG, JPG):**
   - Сейчас: только OCR. Добавить опциональный вызов **vision API** (Gemini Vision / GPT-4o) для генерации аннотации «что на картинке» (схема сети, прибор, диаграмма и т.д.), сохранять в метаданных вложения и в чанке.
   - Точка расширения: `attachment_parsers.extract_text_from_image()` — возвращать также `annotation` от vision; в `confluence_kb_generator` при формировании секции вложения сохранять `metadata.images[]` с `url` (ссылка на вложение Confluence или статику), `description`, `alt`.
2. **Draw.io:**
   - Парсер уже даёт текст подписей. Добавить:
     - вызов LLM (или правило по первому абзацу) для **краткого описания схемы** (1–2 предложения);
     - сохранение в метаданных секции: `metadata.diagram_url` (если будет раздача файла), `metadata.diagram_description`, тип `diagram`.
   - В Qdrant в payload чанка класть `metadata.images` или `metadata.diagrams` с url и description.
3. **Видео/презентации:** если в Confluence есть ссылки на видео или PowerPoint:
   - хранить в `metadata.documents[]` (url, title, description);
   - опционально: аннотация по метаданным или по субтитрам/слайдам (отдельная задача).

**Отладка:**

- Включить vision для одного тестового изображения — проверить, что аннотация попадает в JSON и в Qdrant.
- Для draw.io: один файл .drawio → в документе секция с `metadata.diagram_description` и при поиске по описанию чанк находится.

## 4. Фаза B: Консультация — выдача ссылок на картинки и документы

**Цель:** при ответе Спутникового ассистента не только текст и ссылки на страницы Confluence, но и явный блок «релевантные медиа»: картинки (url + подпись), документы (url + название).

### 4.1. Хранение медиа в чанках Qdrant

- В `kb_loader.py` при создании точек из Confluence-документов передавать в payload:
  - `images: [{ "url": "...", "description": "...", "alt": "..." }]` из секций, где были обработаны изображения/draw.io;
  - `documents: [{ "url": "...", "title": "...", "description": "..." }]` если появятся такие метаданные.
- URL картинок: либо прямая ссылка на вложение Confluence (`/rest/api/content/.../download` или публичная ссылка), либо копирование в статическую папку приложения и URL вида `/static/kb_media/...`.

### 4.2. RAG-ответ: сбор медиа из релевантных чанков

- В `satellite_librarian_agent.py` (или в общем месте RAG для спутникового ассистента): после поиска по запросу по Qdrant для топ‑k чанков собрать:
  - `images = []` из `payload.get("images", [])`;
  - `documents = []` из `payload.get("documents", [])`.
- Передать агенту в контексте блок «Релевантные медиа: изображения и документы» (список url + подписи) и инструкцию в системном промпте: «Если в контексте есть релевантные медиа, упомяни их в ответе и приведи ссылки; не вставляй длинные URL в текст, используй короткие подписи».
- В Streamlit в блоке ответа ассистента: под сообщением отображать блок «Изображения» / «Документы» с превью (img) и ссылками (открыть в новой вкладке, скачать).

**Отладка:**

- Добавить в один тестовый чанк в Qdrant вручную `images: [{ "url": "https://...", "description": "Тест" }]`, выполнить поиск по запросу, убедиться, что в ответе ассистента появляется этот блок и ссылка.
- Проверить, что Confluence-вложения доступны по URL (или настроить раздачу через статику/прокси).

## 5. Фаза C: Образцы draw.io в KB и выдача ссылок

**Цель:** типовые схемы сети (iDirect, Kingsat, топологии) хранить в KB как отдельные сущности с метаданными; ассистент по запросу может предлагать «подходящую схему» и давать ссылку на файл или на страницу Confluence.

### 5.1. Каталог образцов draw.io

- Выделить в Confluence (или в файловой системе проекта) **папку/пространство «Образцы схем»**.
- При синхронизации Confluence страницы с вложениями .drawio из этой папки индексировать с типом `sample_diagram` и метаданными:
  - `title`, `description` (из страницы или сгенерировано LLM по подписям draw.io),
  - `file_url` (скачивание .drawio),
  - `source_url` (страница Confluence).
- Альтернатива: отдельный JSON-каталог `kb_billing/diagram_samples.json` с полями: id, title, description, file_path_or_url, tags (например: iridium, idirect, kingsat, topology).

### 5.2. Поиск и выдача образцов

- При запросе типа «покажи типовую схему подключения iDirect» поиск в Qdrant по чанкам с типом `sample_diagram` (или по тегам) возвращает релевантные схемы; в ответе ассистента — ссылка «Открыть схему» (draw.io или Confluence).
- В UI: кнопка/ссылка «Скачать .drawio» или «Открыть в Confluence».

**Отладка:**

- Добавить один образец в каталог, перезагрузить KB, запросить у ассистента «типовая схема» — проверить, что в ответе есть ссылка на образец.

## 6. Фаза D (идеал): Генерация схем сети по требованиям клиента

**Цель:** по текстовому описанию требований клиента (количество терминалов, тип оборудования, топология) ассистент генерирует **схему сети** в формате, пригодном для draw.io (или аналогичном), т.е. выполняет работу присэйла.

### 6.1. Подходы

| Подход | Описание | Сложность |
|--------|----------|-----------|
| **Шаблоны** | Набор шаблонов draw.io (XML) с плейсхолдерами; LLM выбирает шаблон и подставляет параметры (названия узлов, количество). | Средняя |
| **Генерация XML** | LLM (с structured output) генерирует фрагмент mxfile/mxCell draw.io; скрипт собирает .drawio и отдаёт на скачивание. | Высокая |
| **Генерация Mermaid/PlantUML** | Генерация диаграммы в текстовом формате (Mermaid/PlantUML), конвертация в PNG/SVG и выдача ссылки; для draw.io — последующий импорт или конвертер. | Средняя |
| **Внешний сервис** | Интеграция с API, генерирующим диаграммы (если есть). | Зависит от сервиса |

Рекомендуемый порядок: сначала **шаблоны** и **Mermaid/PlantUML** для быстрого результата; затем эксперименты с генерацией draw.io XML при наличии стабильной схемы mxfile.

### 6.2. Шаги (поэтапно)

1. **Сбор требований в диалоге:** промпт ассистента доработан так, чтобы он задавал уточняющие вопросы (количество сайтов, тип модемов, нужна ли резервация и т.д.) и формировал структурированное описание (JSON).
2. **Выбор шаблона:** по описанию выбирается один из заранее подготовленных шаблонов draw.io (или Mermaid); маппинг «тип задачи → шаблон» в конфиге или в KB.
3. **Подстановка параметров:** в шаблоне заменяются плейсхолдеры на значения из описания; для draw.io — скрипт на Python (парсинг XML, замена value в mxCell).
4. **Выдача файла:** сгенерированный .drawio (или ссылка на Mermaid) отображается пользователю; кнопка «Скачать схему» / «Открыть в draw.io».
5. **(Позже)** Генерация mxfile с нуля по схеме данных от LLM — отдельный эксперимент с валидацией XML и тестами на открытие в draw.io.

**Отладка:**

- Юнит-тест: на фиксированном JSON требований проверить, что выбран правильный шаблон и подстановка даёт валидный .drawio.
- Интеграционный тест: в чате описать задачу → ассистент задаёт уточнения → выдаёт ссылку на схему; открыть файл в draw.io и проверить читаемость.

## 7. Step-by-step с отладкой (сводка)
